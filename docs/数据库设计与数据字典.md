# OpenClaw 数据库设计与数据字典（多租户隔离版）

更新时间：2026-03-13

## 1. 设计目标

- 支撑“1个主管理后台 + 15个子业务后台”的统一数据底座。
- 所有核心业务数据按租户（`tenant_id`）隔离，避免跨客户串数。
- 保持现有项目可平滑迁移，不推翻重建。
- 提供标准化数据字典，统一状态、角色、分类等枚举口径。

## 2. 当前数据库落地状态

已完成以下核心表结构与迁移落地（MySQL）：

- `tenants`：租户主表（新增）
- `users`：用户表（已补充 `tenant_id`）
- `tasks`：任务表（已补充 `tenant_id`）
- `module_settings`：模块配置表（改为 `tenant_id + module_key` 唯一）
- `module_feature_records`：模块子功能记录表（已补充 `tenant_id`）
- `data_dictionary`：数据字典表（新增）

## 3. 核心表设计

## 3.1 tenants（租户表）

关键字段：

- `tenant_id`：主键
- `tenant_code`：业务编码（唯一）
- `tenant_name`：租户名称
- `tenant_type`：`internal | enterprise | personal`
- `status`：`active | inactive`
- `isolation_level`：默认 `logical`
- `owner_user_id`：租户拥有者用户ID（可空）

索引：

- `idx_tenants_owner (owner_user_id)`
- `idx_tenants_type_status (tenant_type, status)`

## 3.2 users（用户表）

关键字段：

- `id`：主键
- `tenant_id`：租户ID（外键到 `tenants.tenant_id`）
- `contact`：登录账号（唯一）
- `role`：`admin | customer`
- `enabled_modules_json`：开通模块列表
- `token_state` / `token_version`：会话控制

索引：

- `idx_users_tenant (tenant_id)`

## 3.3 tasks（任务表）

关键字段：

- `task_id`：业务唯一ID
- `tenant_id`：租户ID（外键）
- `owner_id`：用户ID（外键到 `users.id`）
- `module_key`：子模块编码（15业务之一）
- `status`：`review | queued | running | completed | failed`
- `report_url` / `report_file`：报告定位

索引：

- `idx_tasks_owner_module_updated (owner_id, module_key, updated_at)`
- `idx_tasks_tenant_module_status (tenant_id, module_key, status, updated_at)`
- `idx_tasks_status_created (status, created_at)`
- `idx_tasks_report_file (report_file)`

## 3.4 module_settings（模块配置表）

关键字段：

- `tenant_id`
- `module_key`
- `config_json`
- `updated_by`

关键约束：

- 唯一键 `uniq_module_settings_tenant_module (tenant_id, module_key)`

## 3.5 module_feature_records（模块子功能记录表）

关键字段：

- `record_id`：唯一记录ID
- `tenant_id`
- `task_id`
- `module_key`
- `feature_key`
- `payload_json`

关键约束与索引：

- 唯一键 `uniq_task_feature (task_id, feature_key)`
- `idx_feature_tenant_module (tenant_id, module_key, feature_key, updated_at)`

## 3.6 data_dictionary（数据字典表）

关键字段：

- `dict_type`：字典类型
- `dict_key`：字典键（类型内唯一）
- `dict_value`：字典值
- `dict_label`：展示名
- `description`：说明
- `sort_order`：排序
- `is_system`：系统内置标识
- `status`：启用状态

关键约束：

- 唯一键 `uniq_dict_type_key (dict_type, dict_key)`

## 4. 多租户隔离策略

## 4.1 数据层隔离规则

- 任务、模块配置、模块子功能记录、用户等核心表均带 `tenant_id`。
- 管理端统计/列表/详情接口统一按“当前登录管理员 `tenant_id`”过滤。
- 模块配置读取与写入按 `tenant_id + module_key` 隔离。
- 子模块特征记录查询按 `tenant_id + module_key + feature_key` 过滤。

## 4.2 代码层隔离点

已在以下链路完成租户透传：

- 注册创建用户：生成并落库 `tenant_id`
- 任务创建：任务写入带 `tenant_id`
- 管理端查询：`stats/users/tasks/module/*` 全部带租户过滤
- 模块配置：`find/upsertModuleSettings` 改为租户级
- 子功能记录：`upsert/list/count` 改为租户级

## 4.3 子模块隔离口径

15个子后台以 `module_key` 区分业务域，在同租户内按模块隔离；跨租户再由 `tenant_id` 强隔离，形成“双维隔离”：

- 维度1：`tenant_id`（客户隔离）
- 维度2：`module_key`（子后台业务隔离）

## 5. 数据字典设计（首版）

当前内置字典类型：

- `role`：账号角色
- `task_status`：任务状态
- `tenant_type`：租户类型
- `tenant_status`：租户状态
- `module_category`：模块分类（企业/个人）
- `setting_mode`：模块执行模式（`auto|manual|hybrid`）

配套接口：

- `GET /api/admin/dictionary?type=`

## 6. 迁移与兼容

- 采用“存在即跳过”的增量迁移策略，兼容已有历史库。
- 对历史记录自动补全 `tenant_id`（默认 `t-platform` 或按用户回填）。
- 旧 `module_settings(module_key)` 唯一索引迁移为 `tenant_id + module_key` 唯一索引。

## 7. 后续建议

- 增加 `tenants` 管理接口（创建/停用/套餐绑定）。
- 在 `contact` 唯一策略上评估是否升级为“租户内唯一”。
- 对敏感字段（如用户密码相关字段）在管理端接口输出中彻底脱敏。
