# Task Plan - 全域反诈守护生产可用冲刺（2026-03-13 至 2026-03-19）

## Goal
在不接入抖音/快手 API 的前提下，将当前可用 MVP 提升到可稳定内测/小规模上线版本：
- 修复关键一致性问题（模块 key、权限）
- 补齐基础测试与回归验证
- 强化运行稳定性与运维可用性

## Scope
- In:
  - 反诈与模块工作台的稳定性修复
  - 前后端契约一致性
  - 权限与会话逻辑纠偏
  - 最小必要测试与发布检查
- Out:
  - 平台官方 API 接入（抖音/快手）
  - 支付系统
  - 大规模数据抓取能力

## Phases
- [x] Phase 1 (P0, D1): 统一前后端模块标识与路由契约
  - 对齐 `src/config/modules.ts` 与 `server/src/config/catalog.js` 的 `moduleKey`
  - 增加兼容映射，避免旧数据不可用
  - 验收：模块页不再出现 key 不匹配导致的 404/权限异常
- [x] Phase 2 (P0, D2): 修复前端权限过宽逻辑
  - 修复 `src/stores/auth.ts` 的模块归一化逻辑，按后端真实权限显示
  - 复核路由守卫与菜单显示一致性
  - 验收：未开通模块不可见/不可访问，已开通模块可正常进入
- [x] Phase 3 (P1, D3-D4): 完善反诈与任务流输入校验、错误处理
  - 强化关键请求参数校验和统一错误码返回
  - 补充关键流程边界检查（空输入、超限、非法状态）
  - 验收：非法输入可稳定返回明确错误，不出现未捕获异常
  - 进度：后端校验 + 前端联调校验已完成
- [ ] Phase 4 (P1, D5): 增加最小回归测试集
  - 后端核心服务/路由测试（auth、module task、anti-fraud）
  - 前端关键交互冒烟测试（登录、模块进入、反诈核心流程）
  - 验收：CI/本地可执行，核心场景有自动化保障
- [ ] Phase 5 (P1, D6): 发布前运维与文档完善
  - 明确环境变量、备份恢复、初始化流程
  - 补齐上线检查清单（合规文案、日志、告警）
  - 验收：新环境可按文档在 30 分钟内拉起
- [ ] Phase 6 (P1, D7): 预发布验收与回归
  - 全链路走查与问题收敛
  - 输出发布说明与已知限制
  - 验收：P0 缺陷清零，P1 仅保留可接受项

## Decisions
- 本冲刺保持“无平台 API”策略，数据来源以手工/半手工为主。
- 先修稳定性与一致性，再补自动化与运维。

## Risks
- 模块 key 变更可能影响历史数据与已有任务记录匹配。
- 权限逻辑收紧后，可能暴露旧账号初始化配置问题。
- 测试补齐会暴露现有边界缺陷，需预留修复缓冲。
